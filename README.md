# Spaceship Titanic Passenger Transport Prediction

This repository contains the code submission for our Kaggle **Spaceship Titanic** project. The task is a tabular binary classification problem: predict whether each passenger was transported, using the structured passenger information provided by Kaggle.

Our final submitted solution is the **k0 pipeline**, an XGBoost-based pipeline with feature engineering, Room Fill imputation, permutation-importance pruning, SMOTE class balancing, Optuna tuning, 20-fold cross-validation, and NumPy RNG snapshots for reproducibility.

Final public leaderboard score:

```text
0.81762
```

## 2. Project Overview

The project follows three experimental phases.

### Phase 1: Controlled baseline comparison

We first compared four model families:

- Logistic Regression
- XGBoost
- CatBoost
- LightGBM

The goal was to compare model classes under a controlled setup instead of immediately optimising for leaderboard score. The baseline comparison used the same 19-feature input design and the same validation strategy where applicable.

### Phase 2: CatBoost and SMOTE compatibility analysis

CatBoost gave the strongest isolated baseline result, so we attempted to combine it with SMOTE. This created a structural issue:

- CatBoost benefits from native categorical feature handling.
- SMOTE requires a fully numeric feature matrix for nearest-neighbour interpolation.

Encoding categorical variables before SMOTE removes CatBoost's native categorical advantage. This motivated the final pivot from CatBoost to XGBoost.

### Phase 3: Final k0 XGBoost pipeline

The final k0 pipeline uses XGBoost and includes:

- Room Fill group-based imputation
- Cabin decomposition
- OneHotEncoder preprocessing
- SimpleImputer fallback imputation
- permutation-importance feature pruning
- SMOTE class balancing
- Optuna hyperparameter optimisation
- 20-fold cross-validation
- NumPy RNG state snapshots for reproducibility

## 3. Final k0 Pipeline

The final pipeline in `k0-random-run-07.ipynb` performs the following steps:

1. Load the Kaggle train and test files.
2. Concatenate train and test rows only for consistent non-label preprocessing.
3. Apply Room Fill imputation using the group key extracted from `PassengerId`.
4. Engineer features:
   - split `Cabin` into deck, room number, and side
   - create `TotalSpend`
   - retain useful original spend columns
5. Apply imputation and one-hot encoding.
6. Run permutation-importance feature pruning.
7. Reduce the feature set from 27 encoded columns to 15 final training columns.
8. Apply SMOTE to balance the training labels.
9. Run Optuna hyperparameter search.
10. Train the final XGBoost model with 20-fold cross-validation.
11. Generate the Kaggle submission file.

## 4. Reproducibility

The final k0 pipeline uses three NumPy RNG state snapshots:

```text
before_shuffle
before_smote
before_final_model
```

These snapshots control the main stochastic parts of the pipeline:

1. row shuffling before fold construction
2. SMOTE neighbour sampling
3. final XGBoost model training randomness

The reproducibility notebook is:

```text
k0-random-run-13.ipynb
```

It restores the RNG states generated by `k0-random-run-07.ipynb` and reproduces the prediction vector.

Expected reproducibility result:

```text
Same Ratio = 1.0
Different Rows = 0
```

This means every prediction from the reproduction run matches the original k0 prediction.

## 8. Main Experimental Results

### Controlled model comparison

The Phase 1 comparison evaluates Logistic Regression, XGBoost, CatBoost, and LightGBM under the same baseline setup. CatBoost achieves the strongest isolated baseline result, but it is not used in the final pipeline because of its structural conflict with SMOTE.

### Room Fill ablation

The Room Fill contribution was tested by comparing the final k0 pipeline with and without Room Fill while keeping the other pipeline components fixed.

```text
Without Room Fill: 0.81388
With Room Fill:    0.81762
Contribution:      +0.00374
```

### V3 negative result

A separate V3 development branch achieved strong local validation performance but failed to generalise to the public leaderboard.

```text
Local OOF: 0.81824
Public LB: 0.80383
Gap:      -0.01441
```

This negative result is included to show why the final k0 design avoids aggressive local-only optimisation and keeps the threshold fixed at 0.5.

## 9. Data Leakage Note

The train and test files are concatenated row-wise only for consistent non-label preprocessing and feature schema construction.

The target column `Transported` is never used for test-row feature engineering, imputation, encoding, or preprocessing. Final model fitting uses only labelled training rows.

Therefore, the train-test concatenation is used only as an engineering convenience and does not introduce target leakage.
